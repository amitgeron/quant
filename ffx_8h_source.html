20 /*
21  * Format preserving encryption using the FFX algorithm.
22  *
23  * We demonstrate here a simple encryption process derived
24  * from the FFX algorithms, which is effectively a specific
25  * mode of running a verified encryption code. The
26  * algorithm is Feistel cipher in which the S-boxes are
27  * defined by a symmetric encryption algorithm such as
28  * AES or ChaCha20.
29  * See "Ciphers with Arbitrary Finite Domains" by
30  * John Black and Phillip Rogaway, 2001 --
31  * http://web.cs.ucdavis.edu/~rogaway/papers/subset.pdf
32  *
33  * An instantiation of the algorithm is defined by a
34  * series of parameters:
35  * - the context of the symmetric crypto algorithm,
36  * - key used for the symmetric algorithm,
37  * - number of rounds,
38  * - length of the block in bits
39  *
40  * We consider just two symmetric algorithms for now,
41  * ChaCha20 and AES128CTR. In theory, any symmetric algorithm
42  * operating on a 128 bit block would work, and crytographic
43  * hashes producing at least 128 bits of output could also
44  * be used. In practice, ChaCha20 and AES128 cover most of
45  * the use cases.
46  *
47  * The implementation will produce a result for any block
48  * length lower than 256, although values lower than 32 would
49  * not be recommended.
50  *
51  * The block to be encrypted is passed as a byte array of size
52  * (block_length + 7)/8. When the block_length is not a
53  * multiple of 8, the algorithm guarantees that the extra bits
54  * in the last byte are left untouched. For example, if the
55  * block length is 39, the least significant bit of the
56  * fifth byte will be copied from input to output.
57  *
58  * The number of rounds is left as a configuration parameter,
59  * which is constrained to be even by our implementation. The
60  * required number of passes varies with the application's
61  * constraints. The practical minimum is 4 passes. Demanding
62  * applications can use 8 passes, and the practical conservative
63  * value is 10, as specified by NIST for the FF1 variant of
64  * the same algorithm. This choice between 4, 8 or 10 is
65  * based on "Luby-Rackoff: 7 Rounds are Enough
66  * for 2^n(1-epsilon) Security" by Jacques Patarin, 2003 --
67  * https://www.iacr.org/archive/crypto2003/27290510/27290510.pdf
68  *
69  * Encrypting short numbers, by nature, produces a codebook
70  * of limited size. In many applications, the short number is
71  * part of a larger object that is passed in clear text. In that
72  * case, NIST recommends using as much as possible of that clear
73  * text as an initialization vector, used as "tweak" in the
74  * FFX algorithm. See:
75  * https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-38G.pdf
76  */
